# GONYIK 港翼科技官网 PRD

> 版本：v5.0
>
> 更新：2026-07-21
>
> 状态：当前代码与 CMS 的唯一产品基准；本轮修改仅在本地，尚未发布生产。

## 1. 产品目标与边界

官网用于说明港翼如何把材料技术转化为可选择、可验证、可交付的高性能面料与终端应用。

访客需要快速理解：

1. 港翼是高分子材料与面料科技公司，不是服装零售商；
2. RPO-SOTEX 是固纳体系内用于特定复合面料的无氟纳米膜；
3. OTTER、RAYO 服务日常与户外，KAIS 服务特殊职业与任务场景；
4. 每个面料型号有明确定位、测试资料和适用边界；
5. 终端装备展示实际采用的面料，而不是泛化推荐。

不做：电商交易、未经验证的绝对性能承诺、以抽象动效替代材料信息、为桌面与移动端维护两套内容、通用低代码页面搭建器。

## 2. 品牌与产品口径

- 品牌：GONYIK 港翼科技；对外表述为“固纳旗下”。
- 膜产品：统一使用 `RPO-SOTEX`，不再使用历史名称 `RPO-TEX`。
- 系列：
  - 蓝标 OTTER：无氟防水透湿复合面料；
  - 银标 RAYO：轻量、导湿、防晒等非膜或轻结构功能面料；
  - 黑标 KAIS：防刺、消防等特种专业场景材料。
- KAIS 不是 OTTER 的性能或价格升级版；黑标用于表达独立、克制的专业材料平台，不使用警示感较强的红标。
- 性能、认证和应用声明必须对应具体型号、样品、测试方法及有效报告。

|---|---|---|
| `/` | 品牌与能力总览 | Hero、材料技术、面料平台、测试与认证 |

| `/contact` | 询盘 | 联系方式、合作主题、表单 |

## 4. 技术架构

```text
React 19 + React Router + Tailwind
          ↓
src/api/client.ts
          ↓
Express 领域路由
          ↓
db.json 结构化内容 + public/uploads 媒体文件
```

架构原则：

- 前台与 CMS 复用同一领域组件和字段定义；
- 同一类功能只保留一个实现入口，避免平行组件与分叉数据链路；
- 页面组件不直接操作数据库；所有写入经过领域 API；
- 公共 API 不返回内部面料编号；
- 桌面与移动端使用同一数据和媒体资源，仅调整布局；
- 路由按页面懒加载，首页保留首屏同步入口。

## 5. 数据与功能设计

### 5.1 面料系列

CMS 字段：

- 名称；
- Slug；
- 标语；
- 描述；
- 首页卡片背景图；
- 透明系列徽章；
- 排序。

名称与 Slug 必填，Slug 唯一。删除系列时同时删除其面料，并清空终端装备对这些面料的采用关系。

### 5.2 面料型号

CMS 主字段：

| 字段 | 用途 | 规则 |
|---|---|---|
| 对外商品编号 `public_name` | 前台标题，例如 OTTER T70 | 必填、唯一、创建后锁定 |
| 内部编号 `internal_code` | 内部结构与追溯 | 必填、唯一、创建后锁定，不由公共 API 返回 |
| 产品定位副标题 `product_type` | 说明材料类型 | 必填 |
| 性能标签 `features` | 卡片辅助文字 | 最多 3 个，来自统一标签库 |
| 三轴定位 | 日常—专业、轻盈—强韧、柔软—挺括 | 每轴 1–9，可留空；前台不显示数字 |
| 核心性能 `specifications` | 展开卡片后的数据 | 最多 3 项首层展示 |
| 卡片图 `image` | 面料微距或质感图 | 4:3 裁切，WebP |
| 前台显示 `visibility` | 显示或隐藏 | 默认显示 |

兼容字段 `sku_code` 仅作为系统稳定标识和历史 URL 兼容，由服务端按系列自动生成；CMS 不展示、不要求用户填写。`name` 与 `card_summary` 不再作为独立的面料编辑字段。

新增流程：选择系列 → 填写两个编号及产品定位 → 选择标签与三轴 → 可选填写核心性能 → 裁切图片 → 保存。服务端返回中文校验错误，CMS 原位显示。

删除面料时：

- 面料记录删除；
- 终端装备中的对应 `related_sku_ids` 自动清空；
- 已使用编号保留在编号登记表中，不重新分配。

### 5.3 性能标签库

CMS 只维护标签名称。系统生成不可变 Key；历史主题色字段继续兼容旧数据，但不进入当前编辑界面和前台卡片样式。

删除标签时，服务端同时检查 Key、名称和历史别名，并从所有已关联面料的 `features` 或旧摘要字段中移除；不存在其他标签时对应字段置空。操作完成后 CMS 显示受影响卡片数量。

### 5.4 终端装备

分类字段：名称、Slug、描述、排序。已废弃且前台未使用的分类背景图与适配方式字段不再维护。

产品字段：

- 产品名；
- 特点；
- 卡片核心收益；
- 实际采用的一个或多个面料；
- 产品图；
- 前台显示；
- 排序。

卡片关系文案统一为“采用面料”。它表示该终端产品实际使用了这些港翼面料，不表示推荐范围或所有可兼容面料。

产品图片规范：优先上传带透明通道的 PNG 抠图；CMS 以 1:1 裁切并保留透明通道。卡片底色由前端统一使用纯白，不把白底烘焙进图片，便于复用和保持一致。

删除分类会删除其中产品；排序请求只能包含同一分类的产品。

### 5.5 内容状态与可见性

当前 CMS 日常操作只展示“前台显示/隐藏”。历史 `status` 字段保留作归档兼容，不让普通编辑者同时维护两套含义相近的开关。公共接口排除隐藏和已归档内容。

## 6. 组件设计

### 6.1 公共组件

- `PageShell / PageHero / PageSection / SectionHeader`：页面骨架；
- `CatalogCardShell / CatalogCardMedia`：面料和装备卡片的统一外壳与媒体规则；
- `SkuCard`：面料领域卡片及三轴定位；
- `ApplicationCard`：终端装备及采用面料；
- `CatalogCollection / CatalogEndCta`：卡片集合与开发中尾卡；
- `ScrollSpySections / TechnologyDetail`：技术创新连续阅读与导航；
- `ServiceModule`：洗涤、FAQ、联系模块；
- `InlineMarkup`：受控行内内容标记。

不把 `SkuCard` 与 `ApplicationCard` 合并成万能卡片；两者共享外壳，但保留不同领域语义。

### 6.2 CMS 组件

- `FormField / Modal / SaveCancelButtons / ResponsiveAdminList`：统一表单与移动端管理体验；
- `ContentRailEditor`：顺序、显隐、编辑入口与真实卡片预览；
- `ImageCropper`：唯一底层裁切引擎；
- `CroppedImageField`：面料、系列和终端产品统一的选择、预览、裁切、替换与移除入口；
- `SeriesHomeImageEditor`：只保留系列徽章校验和即时保存，背景图复用 `CroppedImageField`。

图片裁切支持 JPEG、PNG、WebP；默认输出 WebP。终端装备指定输出 PNG，以保留透明通道。初始裁切框加载后即可直接确认，不要求用户先拖动一次。

## 7. 页面与响应式规则

### 7.1 首页

- 三个平台固定为蓝标 OTTER、银标 RAYO、黑标 KAIS；
- 徽章使用透明 SVG/PNG/WebP；
- 技术、产品和验证模块保持当前信息层级，不添加无意义的装饰标签。

### 7.2 面料数据库

- OTTER、RAYO 与 KAIS 的分组表达日常/户外与特种场景差异；
- 面料卡片首层只显示商品编号、产品定位、性能标签、三轴和详情入口；
- 三轴只在两端显示词语，不显示轴名、数字或刻度；
- `SkuCard` 使用容器查询：布局根据卡片自身宽度变化，而不是只按视口断点；窄桌面时图、文字、三轴在同一边框内自动纵向排列；
- 移动端三轴位于卡片内容区，详情入口之后不出现游离元素。

### 7.3 终端装备

- 分类导航依次为日常、户外、特种；特种项前使用淡色 `｜`，避免把“专业”误解为极限户外等级；
- 图片区纯白，透明产品图 `object-contain`；
- 采用面料显示对外商品编号，并链接到对应面料卡片。

### 7.4 技术创新

- 六个模块由 CMS 维护，桌面为文字导航与媒体内容组合，移动端为顶部横向吸附导航；
- 桌面与移动端引用同一个 `image_url`，不重复加载媒体；
- 模块顺序、标题、内容、图片和适配方式均由 CMS 控制。

### 7.5 服务支持

- 洗涤保养不显示 `CARE PROTOCOL · 04 STEPS`；
- 删除列表顶部的长黑线；
- 四项步骤使用浅色细分隔的 2×2 模块，移动端自动单列；
- 步骤编号仅作轻量导航，不压过标题与说明。

## 8. UI 规范

- 主色：深海军蓝、冷灰、白；膜层青仅作选中、进度和细节强调；
- KAIS 使用黑与中性灰，不使用危险警示红；
- 公共页面使用直角、细边框、低对比底色和克制阴影；
- 卡片标签为灰蓝色辅助文字，不使用彩色评分或认证式方框；
- 控件点击区域不小于 40–44px；
- 动效只服务入场、切换、展开和阅读位置；遵循 `prefers-reduced-motion`；
- 不使用自动轮播、粒子、3D 倾斜或与材料无关的 IT 风动效。

## 9. 数据完整性与安全

- 管理接口均要求登录令牌；公共 API 不暴露内部编号、SMTP 密码和管理数据；
- 系列、分类 Slug 唯一；面料两个业务编号唯一且锁定；
- 排序接口验证 ID 完整性，并禁止跨系列或跨分类排序；
- 删除标签、面料、系列时同步清理下游关联；
- 上传文件写入统一媒体登记表，业务记录只保存 URL；
- 被业务内容引用的媒体不可从媒体库直接删除；
- CMS 写入和公共 API 使用 `no-store`，避免成功保存后仍读取旧配置；静态构建资源与上传媒体使用长期缓存。

## 10. 性能方案

- 非首页路由和 CMS 页面按路由拆包；
- 非首屏图片 `loading="lazy"`、`decoding="async"`；
- 面料图裁切并限制最大宽度，默认 WebP；装备抠图使用压缩 PNG；
- 同一时刻的重复 GET 请求合并，完成后不保留可能导致 CMS 陈旧的内存响应缓存；
- Footer 接近视口时再加载详细数据；
- 桌面与移动端不重复请求媒体；
- 发布前必须执行 `npm run build`，关注主包、图片尺寸和路由分包变化。

## 11. 部署与数据保留

本地开发：

```text
npm run dev
npm run build
```

生产发布流程：本地构建 → Git 提交与推送 → 服务器拉取主分支 → 依赖变更时安装 → 生产构建 → PM2 平滑重载 → `/api/health` 健康检查。

`db.json`、`public/uploads`、日志和部署凭据不进入 Git。发布不得覆盖生产数据库与上传目录。本轮明确不执行生产发布。

## 12. 验收基准

发布前至少验证：

1. `npm run build` 无 TypeScript 和构建错误；
2. 面料可新增、编辑、显隐、排序、删除，错误信息可见；
3. 新增面料只要求一个对外商品编号和一个内部编号；
4. 删除性能标签后，所有面料卡片对应标签同步清空；
5. 面料和装备上传均能进入裁切，装备 PNG 透明通道保留；
6. 终端装备“采用面料”链接正确；
7. 删除面料或系列后不存在失效装备关联；
8. 1440px、1024px、768px、390px 下卡片无溢出，三轴始终在卡片边框内；
9. 终端分类的特种项前有淡色分隔；
10. 首页 KAIS 标题、徽章和说明统一为黑标；
11. 洗涤保养无旧英文步骤标识和顶部黑线；
12. CMS 保存后刷新公共页面可立即读取新数据；
13. 生产发布必须另行明确授权。
